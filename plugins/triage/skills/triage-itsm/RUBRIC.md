# Triage Rubric

What each per-ticket analysis agent decides, and how. The agent is **read-only** — it never writes to
Jira; it returns a verdict that the Apply step writes. Be calibrated and honest: a wrong `Agent-ready`
wastes a phase-2 run; an over-cautious `Human-only` buries fixable work.

## Per-ticket procedure (each analysis agent)

1. **Read the ticket** — `acli jira workitem view <key> --fields '*all'`. Note summary, description,
   reported symptoms, reporter, priority, `components`, and any **attachments** (HAR, screenshots, logs).
   **If there are attachments, download and analyze them yourself — see [ATTACHMENTS.md](ATTACHMENTS.md).**
   acli cannot download attachments; use the REST API, parse HARs for the failing request (don't read
   the whole file), and view screenshots with the Read tool. They usually pin the failing
   endpoint/screen. Never ask the reporter to review an attachment that's already on the ticket.
2. **Read linked work** — if the ticket links a `BAC`/`WEB`/`FA` issue, read it too (it may already
   hold root-cause notes). Parse prior `> *...generated by AI during triage.*` comments so you don't
   re-ask resolved questions.
3. **Locate the area** — infer `AI: Area` from symptoms/components, then explore that repo
   (`../api`, `../web-admin`, `../flutter-app`) **read-only** to find the relevant code.
4. **Attempt a lightweight root-cause hunt / repro** — trace the code path; for the API, read models/
   views/serializers; for clients, the screen/bloc/component. You are NOT fixing — you are judging how
   findable the root cause is and how big the fix looks. Record a `rootCauseHypothesis` and `reproResult`
   (`confirmed` / `failed` / `insufficient-info`).
5. **Classify + score** — set the six fields below.
6. **Draft the comment** — triage comment with reasoning + repro result; for `Agent-ready`, an agent
   brief; for `Needs-info`, specific questions. Templates below.

## `AI: Agent Suitability` — the core axis

How autonomously an AFK coding agent can resolve this, end-to-end, via the existing `handle-jira-issue`
flow (branch → fix → test → PR). Generic across bugs and stories.

| Value | Use when |
|---|---|
| **Agent-ready** | Root cause is locatable in code (or the story is fully specified), the fix is bounded, and a regression test is feasible. An agent can plausibly finish with no human decision. |
| **Agent-assisted** | An agent can do most of it, but a human checkpoint is required: a product/design decision, ambiguous expected behavior, a schema/data migration judgment call, or external-system access. |
| **Human-only** | Needs human judgment, manual reproduction/QA, ops/infra or third-party action, or the cause is outside the codebases. |
| **Needs-info** | Cannot be assessed until the reporter supplies more detail (no repro steps, missing account/club/env, "doesn't work" with no specifics). |
| **Not actionable** | Invalid, duplicate, already fixed, won't-fix/declined, OR needs a human product/design/strategy decision before any development can be scoped (not merely more info). |

Tie-breakers: prefer `Needs-info` over a low-confidence guess. Prefer `Agent-assisted` over `Agent-ready`
when any human decision is genuinely required — do not mark `Agent-ready` to be optimistic.

**`Needs-info` dominates.** If you can't assess the ticket without more reporter info, the verdict is
`Needs-info` (and `needsInfo = true`) — never combine it with `Human-only`/`Agent-*`. The apply step
treats `needsInfo = true` as authoritative and routes to *Additional information required* regardless
of the suitability tier, so a contradictory verdict would still be held for info (but don't rely on
that — classify cleanly).

### → Workflow status & components

Each verdict drives a Jira workflow transition in the apply step (see WRITE-RECIPE.md):

| Suitability | Target status | Components set? |
|---|---|---|
| `Agent-ready` / `Agent-assisted` / `Human-only` | **Pending development** | **yes** |
| `Needs-info` | **Additional information required** | no |
| `Not actionable` | **Escalated** | no |

For any ticket headed to **Pending development**, set `components` to the actual implementation
surface — a subset (combination allowed) of: **Backend** (api), **Web Admin** (web-admin), **Flutter**
(flutter-app), **Public Website**, **Design**. Pick what the fix really touches, not just the symptom's
surface (e.g. a web-admin symptom whose root cause is in the API is `Backend`, possibly `Backend` + `Web Admin`).

## `AI: Value` — impact of resolving (High / Medium / Low)

`severity × reach`. **High**: payments/money, data loss/corruption, security, login/onboarding blocked,
or affects many/all clubs. **Medium**: a real workflow is degraded but has a workaround, or affects a
subset. **Low**: cosmetic, rare edge case, or single-user nuisance.

## `AI: Effort` — size of the fix (High / Medium / Low)

**Low**: localized, one or few files, clear change, easy test. **Medium**: spans a couple of modules or
needs care around data/migrations. **High**: cross-cutting, architectural, uncertain blast radius, or
hard to test.

## `AI: Confidence` — trust in this assessment (High / Medium / Low)

How sure you are about suitability + root cause. **High**: root cause confirmed in code / repro reproduced.
**Medium**: strong hypothesis, not fully confirmed. **Low**: educated guess. Phase-2 should re-verify
Low/Medium before committing effort.

## `AI: Area` — routing hint

`API` / `Web Admin` / `Flutter App` / `Multiple` / `Unknown`. Drives which repo phase-2 targets.

## `AI: Priority Score` (optional number)

Rough value-vs-effort rank for `ORDER BY`. Map H/M/L→3/2/1 and compute `round(100 * value / effort)`
(so High-value/Low-effort ≈ 300, Low-value/High-effort ≈ 33). Only meaningful for `Agent-ready` /
`Agent-assisted`; leave unset otherwise.

## Scope & lifecycle

Only **`status = "Open"`** tickets are triaged. Before analysis they are claimed into **Under
investigation** (race-safety: a concurrent run won't re-grab them). After analysis the apply step
transitions each to its target status above. When support later answers an `Additional information
required` ticket they move it back to **Open**, which re-enters scope on the next run. Give every Open
ticket a full classification: attempt repro, and if you can't repro or key facts are missing, lean
`Needs-info` with specific questions rather than guessing.

## Comment templates

acli posts **plain text** (no markdown rendering), so these read as plain text — keep them clean.
Every comment starts with the disclaimer line. Reporters are the internal support team (not end-users).

### Triage comment (every triaged ticket)

```text
> *This was generated by AI during triage.*

AI TRIAGE
Agent suitability: <tier> | Value: <H/M/L> | Effort: <H/M/L> | Confidence: <H/M/L> | Area: <area>

Assessment: <2–4 sentences: what this is, suspected root cause, why this suitability tier.>

Reproduction: <confirmed | failed | insufficient-info> — <how you tried / what you saw>
```

### Append for `Agent-ready` — agent brief

Follow the durability rules in the generic `triage` skill's `AGENT-BRIEF.md`: behavioral, concrete
acceptance criteria. **Do NOT include file paths or line numbers anywhere in the comment** — name the
classes, methods, types, and behavioral contracts instead (e.g. "`UserPayment.get_outstanding_amount()`"
not "`payments/models/user_payment.py:1887`"). Paths and line numbers go stale; symbol names don't.

```text
AGENT BRIEF
Summary: <one line>
Current behavior: <what happens now>
Desired behavior: <what should happen, incl. edge cases>
Key interfaces: <types / functions / config to look for or change>
Acceptance criteria:
- [ ] <testable criterion>
Out of scope: <what not to touch>
```

### Replace the comment body for `Needs-info`

```text
> *This was generated by AI during triage.*

AI TRIAGE — more info needed

What we've established:
- <fact 1>

What we still need (@<reporter>):
- <specific, actionable question 1>
- <specific, actionable question 2>
```

Questions must be specific and actionable — never "please provide more info".
