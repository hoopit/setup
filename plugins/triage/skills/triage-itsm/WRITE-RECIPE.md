# Write Recipe — acli reads + REST writes

> **The canonical writer is [`scripts/apply_verdicts.py`](scripts/apply_verdicts.py).** Don't hand-roll
> field writes — feed it a verdicts JSON file (`--dry-run` to preview). The recipes below document the
> mechanics it implements and are for ad-hoc debugging / reads.

Reads, comments, links → **`acli`** (already authenticated; uses OAuth). Custom-field **values** →
**Jira REST API** (`acli` cannot set them). One consistent token for both field creation and writes.

## Auth

```bash
set -a; . ~/.config/hoopit/jira.env; set +a   # exports JIRA_API_TOKEN, JIRA_EMAIL
JIRA=https://hoopit.atlassian.net
AUTH=(-u "$JIRA_EMAIL:$JIRA_API_TOKEN" -H 'Content-Type: application/json')
acli auth status   # confirm acli side too
```

Never echo the token. Quick check: `curl -s "${AUTH[@]}" "$JIRA/rest/api/3/myself" | jq .accountId`.

## Reads (acli)

```bash
# Discover never-reviewed tickets (primary scope). cf[10981] = AI: Last Reviewed.
acli jira workitem search --paginate --csv --fields "key,status,priority,issuetype" \
  --jql 'project = ITSM AND statusCategory != Done AND cf[10981] IS EMPTY ORDER BY priority DESC, updated DESC'

# Re-review set (edited since last review): fetch reviewed tickets + compare client-side.
acli jira workitem search --paginate --csv --fields "key,status,updated,customfield_10981" \
  --jql 'project = ITSM AND statusCategory != Done AND cf[10981] IS NOT EMPTY'   # keep rows where updated > customfield_10981

# Full ticket detail (per analysis agent)
acli jira workitem view ITSM-1234 --fields '*all' --json
acli jira workitem attachment download ITSM-1234 --output-directory /tmp/ITSM-1234-att   # HAR/screenshots
```

## Field write (REST) — `PUT /rest/api/3/issue/{key}`

Resolve every `customfield_XXXXX` id and select **option id** from [FIELD-MAP.md](FIELD-MAP.md). Select
fields take `{"id": "<optionId>"}` (NOT the label). Build the body in `jq` to avoid quoting bugs:

```bash
KEY=ITSM-1234
NOW=$(date +%Y-%m-%dT%H:%M:%S.000%z)   # e.g. 2026-06-20T14:03:00.000+0200
BODY=$(jq -n \
  --arg suit 10010 --arg val 10020 --arg eff 10031 --arg conf 10040 --arg area 10052 \
  --argjson score 225 --arg now "$NOW" '
  {fields: {
    customfield_AAAAA: {id: $suit},   # AI: Agent Suitability  (option id)
    customfield_BBBBB: {id: $val},    # AI: Value
    customfield_CCCCC: {id: $eff},    # AI: Effort
    customfield_DDDDD: {id: $conf},   # AI: Confidence
    customfield_EEEEE: {id: $area},   # AI: Area
    customfield_FFFFF: $score,        # AI: Priority Score (number; omit if unset)
    customfield_GGGGG: $now           # AI: Last Reviewed (datetime)
  }}')
curl -sf -X PUT "${AUTH[@]}" "$JIRA/rest/api/3/issue/$KEY" --data "$BODY"   # 204 on success
```

Replace the `customfield_*` keys + option ids with the real ones from FIELD-MAP.md. A `400` usually
means a bad option id or a field not on the issue's edit screen (see Field creation → context).

## Comment (acli) — write multiline body from a file

```bash
printf '%s\n' "$COMMENT_BODY" > /tmp/$KEY-comment.txt
acli jira workitem comment create --key "$KEY" --body-file /tmp/$KEY-comment.txt
```

The flag is **`--body-file`** (`-F`) for a file, or `--body` (`-b`) inline. acli posts the body as
**plain text / ADF — it does NOT render markdown**, so write comments to read well as plain text
(headers/blockquotes/`**bold**` will show literally). Every comment still starts with the disclaimer
line `> *This was generated by AI during triage.*` for recognizability (see RUBRIC.md).

## Idempotency

The scope JQL already excludes tickets whose `AI: Last Reviewed >= updated`. Always set `AI: Last
Reviewed = now` in the same `PUT` as the verdict, so a ticket is not re-triaged until it changes again.
On `--dry-run`, write nothing (so nothing is marked reviewed).

# Field creation (one-time setup)

Run once before the first triage. Needs Jira admin. Prefer REST for full control (options + context);
`acli jira field create` works for the bare field but takes no options/context.

For each select field:

```bash
# 1. Create the field
curl -sf -X POST "${AUTH[@]}" "$JIRA/rest/api/3/field" --data '{
  "name": "AI: Agent Suitability",
  "type": "com.atlassian.jira.plugin.system.customfieldtypes:select",
  "searcherKey": "com.atlassian.jira.plugin.system.customfieldtypes:multiselectsearcher"
}'   # -> returns {"id":"customfield_AAAAA", ...}

# 2. Find its default (global) context
curl -sf "${AUTH[@]}" "$JIRA/rest/api/3/field/customfield_AAAAA/context" | jq '.values[]|{id,name,isGlobalContext,isAnyIssueType}'

# 3. Add the options to that context
curl -sf -X POST "${AUTH[@]}" "$JIRA/rest/api/3/field/customfield_AAAAA/context/<CTX>/option" --data '{
  "options":[{"value":"Agent-ready"},{"value":"Agent-assisted"},{"value":"Human-only"},
             {"value":"Needs-info"},{"value":"Not actionable"}]
}'   # -> returns each option with its numeric id
```

Field types: select → `...customfieldtypes:select` (searcher `multiselectsearcher`); datetime →
`...customfieldtypes:datetime` (searcher `datetimerange`); number → `...customfieldtypes:float`
(searcher `exactnumber`). Creating a field with no project/issue-type association yields a **global**
context by default — keep it global so ITSM + BAC/WEB/FA stories all see the field.

After creation, record every field id + option id into [FIELD-MAP.md](FIELD-MAP.md). Then run the
smoke test on `ITSM-2792` (PUT one field, `acli ... view` to confirm) before any batch run.
